## 1. 项目目标

HAM 台网点名主控台的初始目标，是为业余无线电台网点名活动提供一个主控可用的现场记录工具。主控需要在活动过程中快速记录友台呼号、时间、QTH、设备、天线、功率、模式、信号报告和备注，并最终导出 Excel 台网日志。

从构想到 V0.9 的迭代中，产品逐步从“手工记录表单”扩展为“监听源候选 + 呼号资料库补全 + 多平台本地/网络发布”的完整工具。

核心原则：

- 点名记录必须由主控确认后写入，不自动把监听数据直接入表。
- 监听源只提供候选和补全信息，避免误记录。
- 左侧是主控工作区，右侧是实时监听区。
- 本地版优先保证 FMO、MMDVM、HAMBOX 等局域网设备可用；网络版主要提供 BM DMR 模式体验。
- Excel 输出要能直接用于台网活动归档。

## 2. 关键版本演进

### 原型阶段

参考了已有 Windows 台网点名软件的界面和两个 `.db3` 数据库，并结合 FMO-Dashboard 的实时仪表盘逻辑，确定了第一版布局：

- 顶部填写台网活动名称、主控呼号、主控 QTH、主控设备、主控功率。
- 左侧主控记录区输入呼号、QTH、设备、功率、模式、信号报告和备注。
- 右侧显示最近通联列表，点选后进入左侧记录区。
- 底部或左侧显示已记录日志，并支持 Excel 导出。

### V0.1.x 阶段

这一阶段主要解决“能用”和“真实数据源可用”：

- 接入 FMO 实时事件流，修正早期误读日志页的问题。
- 增加 MMDVM `lh.php` Last Heard 读取。
- 增加 BrandMeister BM DMR 通话组监听。
- 增加 HAMBOX 监听入口。
- 支持已记录日志编辑、删除、搜索、导出。
- 支持网络版保存到 VPS，并增加监控后台。
- 打包 Win64 本地版，解决 Electron 本地服务、窗口新建活动和文件保存逻辑。

### V0.9 阶段

V0.9 是第一个相对完整的测试发布版本：

- 统一图标、标题、关于页、VPS 介绍页和下载入口。
- 支持 Win64 本地版、macOS Universal DMG 和 VPS 网络版。
- 引入呼号数据库启用/注册/密钥机制。
- 呼号、QTH、设备、天线、功率、模式、信号报告支持数据库补全。
- Excel 输出改为 `.xlsx`，增加台网活动信息、主控信息、天线列和页脚。
- 顶部增加秒级系统时间显示。
- 已记录数量可设置，下一条记录序号从设置值 +1 开始。
- 记录区选择列调整到右侧，并恢复全选删除。
- 网络版公开说明页提供截图、主要功能、下载链接、GitHub 项目和网络版体验入口。

## 3. 当前主要功能

### 台网活动信息

顶部记录：

- 台网活动名称，默认按日期生成。
- 主控呼号。
- 主控 QTH。
- 主控设备。
- 主控功率。
- 保存、自动保存、新建活动。
- 已记录数量和下一条序号设置。
- 秒级系统时间显示。

### 主控记录区

左侧记录区支持：

- 前缀输入，例如输入数字自动转换为 `B数字/` 前缀。
- 主呼号输入，主框不允许斜杠，前缀和后缀由独立逻辑处理。
- 呼号候选卡片。
- QTH、设备、天线、模式、功率、信号报告补全。
- “首次参与”备注提示：当库中没有该呼号时自动提示。
- 已参加次数提示：从呼号数据库和历史记录中推算。
- 添加记录后自动生成时间和序号。

### 实时监听区

右侧监听区统一显示：

- 呼号。
- 时间，显示年月日和分钟。
- QTH。
- 设备名称。
- 功率。
- 模式。

支持的监听源：

- FMO：本地版可稳定使用；网络版受浏览器和网络限制。
- MMDVM：读取本地设备 `mmdvmhost/lh.php`。
- HAMBOX：读取本地设备 Dashboard/Last Heard 数据。
- BM DMR：通过 BrandMeister 网络监听指定通话组。
- YSF、FCS、D-Star / XLX、P25、NXDN：保留入口，未开放实时监听。

网络版限制：

- 网络版主要提供 BM DMR 模式测试。
- 局域网设备访问受浏览器安全策略、HTTPS mixed content、CORS 和设备接口影响，稳定接入建议使用本地版。

### 主控发射提示

底部主控发射栏用于提示主控是否正在发射。

最终采用的核心规则：

- 取实时通联列表首位。
- 将首位呼号与主控呼号做核心呼号匹配。
- 支持 `B3/BH1JSS`、`BH1JSS/1DR` 等带前缀/后缀形式。
- 首位匹配主控时高亮提示。
- 下一轮监听发现首位不再是主控时恢复普通状态。
- 对没有明确结束事件的来源，使用短时间失活兜底，避免高亮长时间残留。

这个规则比绑定具体协议事件更适合多监听源，因为 FMO、BM、MMDVM、HAMBOX 对“正在发射”的表达方式并不一致。

### 已记录区与 Excel

已记录区支持：

- 倒序显示，最后添加的记录在最上方。
- 序号随设置的已记录数量继续递增。
- 搜索呼号、QTH、设备、备注。
- 全选和批量删除。
- 记录编辑。
- Excel 导出。

Excel 输出要求：

- 文件扩展名为 `.xlsx`。
- 文件名按活动名称自动生成。
- 首行显示活动名称和主控信息。
- 包含天线列。
- 页脚显示：`本日志由 HAM台网点名主控台 自动生成，技术支持BH1JSS`。
- 尽量兼容 Excel 和 WPS。

## 4. 呼号数据库与同步机制

### 基础库来源

早期使用参考 Windows 点名软件的两个 `.db3` 数据库提取历史资料，按呼号合并去重，生成基础库。基础库字段包括：

- 呼号。
- QTH。
- 设备。
- 天线。
- 功率。
- 模式。
- 信号报告。
- 参与时间。

后续每次点名活动保存记录时，如果出现新的呼号、QTH、设备、天线、功率、模式等资料，可以进入呼号资料库。

### 本地库与共享库

V0.9 前后讨论并实现了“启用呼号数据库”的高级功能：

- 默认本地版不再把完整基础库作为强依赖。
- 用户需要提交注册信息，包括呼号、CRAC 操作证书号、常用 QTH、常用服务器等。
- 管理后台审核后发放密钥文件。
- 本地版导入密钥文件后启用共享呼号数据库。
- 密钥文件不是明码形式，支持重装或换机后重新导入。
- 启用后可同步网络端资料库。

注册和启用后的 UI 文案最终确定为：

- 开关文案：`启用呼号数据库`。
- 启用后状态显示：`共享 xxxx 条，新增 x 条`。
- 按钮：`同步`、`注册`。

### 补全逻辑

补全逻辑经历了几轮调整，最终规则如下：

1. 呼号输入优先。
2. 输入呼号后，优先检索当前呼号下的历史资料。
3. QTH、设备、天线、模式、功率、信号报告的候选，优先来自当前呼号。
4. 如果用户在某个字段手动输入片段，则可在全库内模糊匹配。
5. 例如当前呼号 `BH1JSS` 曾使用 `DM-32UV`，默认候选优先显示该呼号历史设备；如果用户输入 `7`，则可出现全库中包含 `7` 的设备。
6. 呼号框本身也需要从呼号库中模糊检索和补全。
7. 不在库中的呼号，备注默认提示 `首次参与`。

## 5. 实时通联读取方案

本项目的实时通联读取逻辑是从 FMO-Dashboard 演进而来，但做了多源扩展。

FMO-Dashboard 主要围绕 FMO 事件和日志：

- `speakingHistory` 维护正在发言和刚结束发言。
- `activeContact` 表示当前发言台。
- `displayRecords` 合并 speaking history 和日志记录。

HAM 台网点名主控台的改进：

- 不把日志页当作实时列表。
- FMO 首次连接后从实时事件逐渐累积列表。
- 各来源统一归一化成 Candidate 模型。
- BM DMR 过滤初始化历史包，避免误触发主控发射。
- MMDVM 和 HAMBOX 的 Last Heard 不能永久视为正在发射。
- 实时列表首位驱动候选和主控发射提示。

详见：`docs/realtime-qso-feed-design.md`。

## 6. 构建与发布现状

### Web / VPS

网络版服务结构：

- 前端：`npm run build` 输出 `dist/`。
- 服务端：`server/index.mjs`。
- 线上挂载：`/checkin/`。
- 数据目录：`/var/lib/ham-checkin`。
- 监控后台：`/checkin/admin/monitor`。
- 公开介绍页：`https://fmo.bh1jss.net/`。

网络版限制：

- 仅 BM DMR 作为主要测试监听源。
- 本地 FMO/MMDVM/HAMBOX 能否访问取决于用户浏览器和局域网环境。

### Windows

当前使用 Electron 打包 Win64 本地版：

- 主入口：`electron/main.mjs`。
- 应用内启动本地 Node 服务。
- 页面通过本地服务运行，而不是简单 `file://`。
- Win64 版本包含运行环境，因此体积较大，约 90MB 级别。

常用命令：

```bash
npm run build
npm run dist:win
```

注意：

- 当前 Win64 版本不支持 Win10 32 位和 Win7 x86。
- 如需 32 位版本，需要单独评估 Electron/Node/依赖对 ia32 和 Win7 的支持。

### macOS

macOS 版本使用 Electron Builder 生成 Universal DMG：

```bash
npm run dist:mac:universal
```

重要卡点：

- 普通本地 DMG 可以安装，但从网页下载后会触发 Gatekeeper 校验。
- 解决方式不是分 Intel/Apple Silicon 架构，而是 Developer ID 签名 + Apple Notarization + staple。
- 后续接入 `Developer ID Application` 证书和 App Store Connect notarization 后，网页下载 DMG 才能避免“无法验证开发者/包含恶意软件”拦截。
- V0.9 已经围绕 macOS 签名、公证和 Universal 包做过工作，但正式分发需要持续确认 CI 密钥和证书状态。

## 7. 重要工程卡点

### 7.1 FMO 实时列表误读日志页

早期问题：

- FMO 地址连接后，首位能显示正在通联，但后续列表常常来自日志页。
- 这导致列表看起来有 20 条，但并不是真正实时积累。

修正方向：

- 按 FMO-Dashboard 仪表盘逻辑读取实时事件。
- 首次打开时允许列表为空，随着实时事件逐步累积。
- 日志和 API 只用于补全，不用于伪装实时列表。

### 7.2 监听源切换失败

问题表现：

- 从 BM DMR 切换到 FMO 或 MMDVM 后出现 `Failed to fetch`。
- 本地网页、macOS 版和网络版都曾出现类似现象。

处理方向：

- 切换监听源时彻底停止旧监听连接和旧轮询。
- 清理旧 AbortController、WebSocket、定时器和 pending fetch。
- 切换后重新初始化当前来源的状态。
- 候选残影要清空，避免点击新呼号后仍带入上一个呼号的 QTH/设备。

### 7.3 主控发射提示不能及时恢复

问题表现：

- FMO 下可以收到结束事件，恢复较好。
- BM/HAMBOX/MMDVM 依赖轮询或 Last Heard，经常要等下一个呼号出现才恢复。

处理方向：

- 统一使用“实时列表首位核心呼号是否匹配主控呼号”作为 UI 规则。
- 来源有明确结束事件时立即恢复。
- 没有明确结束事件时使用 1 秒级失活兜底。

### 7.4 Excel 扩展名和兼容性

问题表现：

- 早期导出扩展名为 `.xls`，部分软件提示扩展名和文件格式不匹配。
- 部分表格软件不显示网格线。

处理方向：

- 改为真正的 `.xlsx`。
- 使用 ExcelJS 生成工作簿。
- 输出加入标题、主控信息、数据表头、页脚。
- 尽量设置边框和样式，而不是依赖软件默认网格线。

### 7.5 Win64 新建活动

问题表现：

- Web 版可以通过打开新标签页实现新建活动。
- Win64 桌面版没有浏览器标签页概念，新建活动不能沿用网页逻辑。

处理方向：

- 新建活动改为重置当前本地活动状态。
- 新活动已记录数量从 0 开始。
- 保存逻辑生成新的日志文件。

### 7.6 macOS Gatekeeper

问题表现：

- 本地生成的 DMG 可安装，但从网页下载后无法打开。
- 分架构包不能解决 Gatekeeper 拦截。

处理方向：

- 需要 Developer ID Application 证书签名。
- 需要 `notarytool` 提交 Apple 公证。
- 公证通过后需要 `stapler staple`。
- 最终再上传到 VPS 和 GitHub Release。

### 7.7 VPS 下载页与公开入口

问题表现：

- GitHub Release 有产物，但 fmo.bh1jss.net 首页没有清楚的软件介绍和下载入口。
- 下载链接曾指向旧版本。

处理方向：

- 首页增加台网点名主控台介绍区。
- 使用真实截图，压缩但不裁切。
- 明确网络版体验、Win64 下载、macOS 下载、校验文件和 GitHub 项目。
- 下载文件同步到 VPS 本地下载目录，避免只依赖 GitHub。

## 10. 后续建议

### 产品侧

- 注册审核流程可以进一步增加状态查询，避免用户不知道是否已审核。
- 呼号数据库可增加冲突合并策略，例如同一呼号多个 QTH/设备的可信度排序。
- Excel 模板可以增加多种格式：正式台网日志、简洁签到表、活动统计表。
- 右侧监听源中 YSF/FCS/D-Star/P25/NXDN 可逐步接入真实网络监听接口。

### 工程侧

- 抽离 `src/App.vue` 中过多 UI 和业务逻辑，逐步拆成 composable/store/service。
- 为监听源切换增加自动测试或至少可复现的手工 checklist。
- 给呼号数据库补全规则增加单元测试，尤其是“当前呼号优先 + 全库模糊兜底”。
- 将 VPS 首页和下载页纳入仓库管理，减少手工编辑服务器静态文件。
- 为 Win64 和 macOS 构建产物建立统一 release checklist，包括 checksum、公网下载验证、安装验证。

### 发布侧

- macOS 正式下载包应保持 Developer ID 签名、公证、staple 三步完整。
- Win64 如需减少体积，需要评估是否提供 portable、installer 和 web-only 三种形态。
- 网络版继续明确“BM DMR 测试体验”的边界，避免用户误以为公网服务器能稳定访问自己的局域网设备。

## 11. 给后续项目的可复用经验

1. 先把真实业务流程切开：监听候选、人工确认、日志记录、资料补全是四件事。
2. 多来源实时数据必须统一模型，否则 UI 会被协议细节污染。
3. Last Heard 不是实时发射状态，不能简单当成 speaking。
4. 监听源切换要彻底清理旧连接，否则最容易出现残影、死锁和 `Failed to fetch`。
5. 本地设备接入不要过度依赖网络版，桌面本地服务更可靠。
6. Excel 交付不是“能下载”即可，要考虑扩展名、样式、表头、页脚和目标软件兼容性。
7. macOS 网页下载分发的核心是签名和公证，不是 DMG 能否本地打开。
8. 发布完成必须同步 GitHub、VPS 下载目录、首页介绍、checksum 和版本文案。

这套经验可复用于后续 HAM 工具、FMO-Dashboard 迭代、局域网设备监听工具和其它多平台桌面/Web 混合项目。
